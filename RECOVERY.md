# Site Source Recovery — Status & Runbook

**Last updated:** 9 July 2026

## The situation (plain English)

The live website (www.greenvac.com.au) is a static-HTML site hosted on
Vercel, project **greenvac-services**. Every deployment in its history was
pushed straight from an AI coding agent's temporary workspace via the
Vercel CLI — Codex sessions in early May 2026, Claude Code sessions from
late May to 1 June 2026. Those workspaces are deleted when a session ends.

**Result: the only complete copy of the site is the deployed snapshot
inside Vercel.** There is no git history, no backup. James checked his
computer and Google Drive: no local copy exists.

This repo exists to fix that. Once recovery is complete, the `site/`
directory in this repo becomes the single source of truth, and the Vercel
project gets connected to this repo so every future change is tracked and
reversible.

## What's blocking recovery right now

The Claude Code cloud environment this session runs in has a restricted
network policy: it cannot fetch from `greenvac-services.vercel.app` or
`www.greenvac.com.au` (the egress proxy returns 403 for those hosts).
The Vercel MCP integration can fetch pages, but it corrupts binary files
(images, fonts, favicons) and silently truncates large files, so it cannot
be used for a faithful mirror.

**The fix is a one-time settings change by James:**

1. Open Claude Code on the web (claude.ai/code) and go to the settings for
   the environment used for this repo's sessions.
2. Under the environment's **network access** settings, either add these
   domains to the allowlist, or switch the environment to full network
   access:
   - `greenvac-services.vercel.app`
   - `www.greenvac.com.au`
   - `greenvac.com.au`
3. Reply in the session (or start a new one on branch
   `claude/greenvac-seo-fix-wpcsvr`) and say the network is open.

Docs on environments and network policy:
https://code.claude.com/docs/en/claude-code-on-the-web

## How recovery runs once the network is open

```
bash scripts/mirror-live-site.sh
```

The script:

- downloads all 30 pages listed in the live sitemap, plus the 404 page,
  `robots.txt`, `sitemap.xml`, CSS, JS, fonts, favicons, and the stray
  build files named in robots.txt (`compress.js`, `compress_all.py`,
  `get-a-quote-src/`);
- scans every downloaded HTML/CSS file for image/font/script references
  and downloads those too, repeating until nothing new appears;
- **verifies every file's md5 checksum against the ETag Vercel serves**
  (for Vercel static files the ETag is the md5 of the content), so we know
  each recovered file is byte-for-byte identical to production;
- writes `mirror-report.txt` listing every file as OK / OK-NOMATCH / MISS.

After the run: review the report, fix any misses, reconstruct
`vercel.json` (see below), commit `site/`, push.

## Files that cannot be mirrored (must be reconstructed)

- **`vercel.json`** — Vercel never serves this file. The live responses
  carry security headers (CSP, HSTS, X-Frame-Options, Referrer-Policy,
  Permissions-Policy, X-Content-Type-Options) which on a static deploy are
  configured in `vercel.json`. The exact header values are captured in
  `.mirror-headers/` by the script; rebuild `vercel.json` from those.
- **`.vercel/project.json`** — links a local folder to the Vercel project;
  regenerated by `vercel link`, never committed.

## After recovery: connecting Vercel to GitHub

Agreed with James (see docs/SEO-FIX-PLAN.md): once `site/` is verified
complete, connect the Vercel project **greenvac-services** to this GitHub
repo in the Vercel dashboard (Project → Settings → Git):

- **Root Directory:** `site` (the repo also holds docs and scripts that
  must not be deployed)
- **Production branch:** `main`

Do NOT connect git before the mirror is verified complete — an auto-deploy
from an incomplete tree would break the live site.

  is leftover boilerplate.
- Both `/services` and `/services.html` return 200 (Vercel default static
  handling); plain `.html` files reproduce live URL behaviour.
- Current production deployment: `dpl_7PDFS7i4n2cWLWoEqqaq8SbkWZgT`,
  created 1 June 2026 via CLI. (Static-file `Last-Modified` headers show
  later dates — those are CDN cache timestamps, not newer deploys.)
